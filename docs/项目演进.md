### 表现
用户发送“你好”
从日志看，总耗时约 19 秒，瓶颈是 3 次串行的 LLM 调用：

阶段	耗时	作用
classify_intent	~7s	判断意图
handle_chat	~12s	生成回复
post_turn_extract	~几秒（含在12s内）	判断是否需要存记忆
对于"你是谁？"这种简单问题，意图分类和记忆提取都是不必要的开销。

从日志时间戳分析：

阶段	耗时	说明
classify_intent	5s	LLM 调用 1
load_memory	<1s	正常
generate_plan	20s	LLM 调用 2
Step 1: calendar	<1s	本地 mock
Step 2: search	54s	LLM 调用 3（最大瓶颈）
Step 3: todo	<1s	本地 DB
Step 4: synthesize	55s	LLM 调用 4
post_turn_extract	7s	LLM 调用 5
总计	~2 分 22 秒	5 次串行 LLM 调用
最大的两个瓶颈是 SearchTool（54s，因为是 LLM 模拟搜索）和 synthesize（55s，汇总长文本）。

### 问题
- 意图分类`llm call 1`花费5s，时间成本不可接受
- **记忆提取**没有异步处理（还有一个核心问题：什么时候需要提取记忆）
- Step 2: search和Step 4: synthesize 太慢了
- 没有给token成本限制
- 没有流式输出
- 没有针对大小模型的多种api的配置
- 没有分成auto/chat/work模式，如果都走一套流程很慢
### 解决
- 意图分类用小模型
- **记忆提取**异步处理（还有一个核心问题：什么时候需要提取记忆）
- 流式输出
- 添加针对大小模型的多种api的配置
- 增加auto/chat/work模式
